### SOLID原则检查清单

#### S - 单一职责原则 (Single Responsibility)
- [ ] 每个类/模块是否有且仅有一个改变的理由
- [ ] 是否避免"上帝类"（承担过多职责的类）
- [ ] 方法是否专注于单一功能

#### O - 开闭原则 (Open/Closed)
- [ ] 类是否对扩展开放，对修改关闭
- [ ] 是否使用抽象和多态代替条件判断
- [ ] 新功能是否通过扩展而非修改实现

#### L - 里氏替换原则 (Liskov Substitution)
- [ ] 子类是否能替换父类而不影响程序正确性
- [ ] 是否正确使用继承关系
- [ ] 覆盖方法是否符合父类契约

#### I - 接口隔离原则 (Interface Segregation)
- [ ] 接口是否保持精简（避免臃肿接口）
- [ ] 是否使用多个专用接口而非一个通用接口
- [ ] 依赖是否建立在最小接口上

#### D - 依赖倒置原则 (Dependency Inversion)
- [ ] 是否依赖抽象而非具体实现
- [ ] 高层模块是否不依赖低层模块
- [ ] 是否使用依赖注入/反转容器

### 代码质量检查

#### 注释完备性
- [ ] 所有公共类和函数都有文档字符串
- [ ] 复杂逻辑有行内注释说明
- [ ] 配置项有清晰的说明
- [ ] 接口方法有参数和返回值说明

#### 命名规范
- [ ] 类名使用 PascalCase
- [ ] 函数和变量使用 snake_case
- [ ] 常量使用 UPPER_SNAKE_CASE
- [ ] 命名具有描述性，避免缩写

#### 错误处理
- [ ] 所有外部调用都有异常处理
- [ ] 自定义异常有清晰的继承结构
- [ ] 错误信息具有描述性

#### 测试覆盖
- [ ] 核心功能有单元测试
- [ ] 边界条件有测试覆盖
- [ ] 错误场景有测试覆盖
